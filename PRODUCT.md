# Product

## Register

product

## Users

Developers working on the Pilota POC: 1-3 engineers on their local machine during active development. They open the hub in a pinned browser tab, glance at it to navigate between services, and need immediate branch awareness (which stack is running). No onboarding needed: they know the project.

## Product Purpose

Internal developer navigation hub for the Pilota SDK proof-of-concept. Lists all running services per branch with their live *.localhost URLs. Lets developers jump instantly between frontends, backends, and tooling. Doubles as a presentation artifact that communicates what each service validates for the Pilota SDK approach.

## Brand Personality

Precise, direct, tool-forward. The design should feel like something the team built for themselves: sharp, confident, no marketing gloss.

## Anti-references

- Generic SaaS status-page dashboards (green dots, grey cards, identical rows)
- Portainer-style heavy data tables
- "DevHub" templates with navy backgrounds and icon grids
- Dashboard UIs that look like they sell monitoring SaaS

## Design Principles

1. Navigation first: every element should reduce the time to reach a service.
2. Branch in the foreground: the current branch must be the most legible thing after the service names.
3. Context without documentation: the Valide lines add SDK rationale without turning the page into a README.
4. Familiar structure, distinctive execution: predictable grid layout, but the visual language is clearly owned.
5. No chrome, no drama: internal tools don't need hero sections or decorative motion.

## Accessibility & Inclusion

WCAG AA minimum. Color is never the sole differentiator for state.
